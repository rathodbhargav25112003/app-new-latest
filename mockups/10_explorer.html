How the shorter flow works (same endpoints)
Mount

One request fans out

On open, fire /category + /subcategory?category_id + /topic?subcategory_id in parallel. All three chip rows + topic sections render together.

 Tap chip

Client-side filter, no re-navigate

Switching category just swaps the subcategory chip row (already cached) and re-fetches topics. No new route, no loading state flash.

 Tap topic

Inline expand

Expanded card fetches /questions?topic_id (or /videos / /notes). First 3 render inline with "see all" trailer.

 Offline

Pre-cache subcategories

We already fetch the full hierarchy at app start. The Explorer just reads from the cached tree — no extra round trips for the chip rows.

Key UX wins
4 → 2 taps

Same destination, half the friction

Average 50% fewer taps to reach a question / video / note. Back button never used.

 Lateral speed

Switch siblings without going up

Chip rows mean jumping from Shock to Wound healing is one tap, not back-back-tap.

 Visible state

"What have I already done"

Progress rings on topic chips (42% attempted) and per-MCQ icons (✓ done / ✗ wrong / ○ untouched) eliminate ambiguity.

 Smart defaults

The student's last subcategory pre-selected

On reopen, the chip row remembers where they were — one tap to resume context.

 Bulk actions

"Drill all · Wrong only · Unseen only"

Topic headers expose batch practice buttons so students don't drill one question at a time when they meant 20.

 Zero API changes

Same endpoints as today

Uses existing category / subcategory / topic / questions endpoints. Only the client structure changes.
